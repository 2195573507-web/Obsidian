# 2026-06-28 OpenClaw 技术晚间补充

> 范围：截至 2026-06-28 11:20 UTC 前约 24 小时内检索到的 AI / 开源 / 安全 / 运维动态。只收录对 OpenClaw、Hermes、个人服务器或自托管运维较重要的变化；来源不足处标注“未确认”。

## 最重要 5 条

1. **Langflow CVE-2026-5027 被报道出现野外利用，公开暴露实例风险高。** The Hacker News 引用 VulnCheck 称，Langflow 的 `POST /api/v2/files` 路径遍历/任意文件写入漏洞已被利用；NVD 与 Tenable 均确认漏洞本质为 `filename` 未净化导致任意路径写入。Tenable 建议升级到 **Langflow 1.9.0+**；报道提到约 7,000 个公网暴露实例。对自托管 AI 工作流尤其重要。
2. **MLflow CVE-2026-13484 新增记录：实验级 Label Schema CRUD API 缺少授权。** NVD 记录显示影响 MLflow 至提交 `4666cff...`，远程可触发、复杂度高、需要低权限，PoC 已公开；同时 GitHub 讨论称相关 PR 尚未合并、发布前会补 auth handler。风险目前更偏“关注/预发布接口治理”，不是立即大规模利用信号。
3. **Kubernetes 官方 CVE Feed 在 6 月 28 日刷新，但未见过去 24 小时新 Kubernetes CVE。** Feed 最新条目仍是 2026-04-10 的 CVE-2026-3865（SMB CSI Driver `subDir` 路径遍历可能删除非预期目录），状态 fixed。结论：K8s 本体当天没有新的高优先 CVE，但已有 ingress-nginx / CSI 历史修复仍值得巡检版本。
4. **Google Security Operations 6 月 26 日更新文档导航，并预告 ingestion metrics 修正。** Google SecOps release notes 显示文档门户按 Discover / Administer / Ingest / Detect / Analyze and respond / Automate 等工作流重组；此前 6 月 23 日还加入 Gemini Cloud Assist 到 Feed Management，并说明 6 月 29 日至 7 月 10 日部分区域 ingestion 指标会出现一次性“表观 spike”，实际日志量不变。
5. **AI 防守工具继续向“多模型/多 agent 漏洞验证”发展。** Microsoft Build 2026 安全公告介绍 MDASH：用 100+ 专用 AI agents、多模型编排发现、验证并证明可利用漏洞；Anthropic Project Glasswing / cyber defender 文章也强调 frontier model 已能发现高严重漏洞并投入防守。虽然并非今天首次发布，但它们与 Langflow/MLflow 动态共同说明：AI 应用基础设施本身与 AI 辅助安全治理正在同时升温。

## 对 OpenClaw / Hermes / 个人服务器有用的 3 条

1. **若运行 Langflow 或类似低代码 AI 编排工具：立即确认版本与公网暴露面。** 建议：升级 Langflow 1.9.0+；关闭 unauthenticated auto-login；放到 VPN/内网；Nginx/防火墙限制 `/api/v2/files` 等上传接口；检查容器/主机是否有异常写入。
2. **把“AI 工具链依赖”纳入常规漏洞巡检。** MLflow、Langflow、Docker Model Runner、MCP/agent 工具这类组件正在变成攻击面；Hermes/OpenClaw 若接入外部工具或工作流服务，应记录版本、暴露端口、认证方式，并在日报中单列 AI 工具链 CVE。
3. **监控指标解释要留意供应商口径变化。** Google SecOps 的 ingestion metrics 修正提醒：日志/费用/告警仪表盘出现 spike 时，先查 release notes 和计量口径变更，不要直接误判为入侵或流量异常。

## 风险 / 机会

- **风险：AI 应用基础设施公网暴露 + 默认弱认证。** Langflow 案例显示，一旦上传、插件、MCP、自动登录等能力组合在公网暴露，漏洞可从“文件写入”升级到 RCE 或横向移动。
- **风险：预发布/实验 API 被安全数据库收录后容易造成噪音。** MLflow CVE-2026-13484 需要区分“已发布生产影响”与“尚未合并功能”的实际风险，避免误报，但也提示开发阶段就要补授权测试。
- **机会：将 AI 用于补丁优先级与漏洞验证。** Microsoft / Anthropic 的方向说明，多 agent、多模型验证可帮助从“理论 CVE 列表”转向“可利用风险排序”；个人服务器可先用轻量方式落地：依赖清单 + CVE feed + 暴露面扫描 + 人工确认。
