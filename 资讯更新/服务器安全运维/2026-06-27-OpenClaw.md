### 1. Kubernetes 官方 CVE Feed 更新：新增/列出多项存储与 ingress-nginx 风险
- **确认度：高**（Kubernetes 官方 feed，更新时间 2026-06-26 23:54 UTC）
- **要点**：官方 CVE Feed 当前列出包括：
  - `CVE-2026-3865`：SMB CSI Driver `subDir` 路径穿越，可能在删除/清理时误删 SMB 服务器非预期目录；修复版本 `>= v1.20.1`。
  - `CVE-2026-3864`：NFS CSI Driver `subDir` 路径穿越，可能误删/修改 NFS 导出目录；修复版本 `>= v4.13.1`。
  - `CVE-2026-4342`：ingress-nginx 注释/配置注入，可导致 controller 上下文任意代码执行与 Secret 泄露；受影响版本为低于 `v1.13.9`、`v1.14.5`、`v1.15.1` 的分支。
- **运维含义**：如果集群允许非可信用户创建 PV，或使用 ingress-nginx，应优先检查版本与 RBAC。
- **来源**：https://kubernetes.io/docs/reference/issues-security/official-cve-feed/ ；JSON feed：https://kubernetes.io/docs/reference/issues-security/official-cve-feed/index.json

### 2. runc 三个高危漏洞可导致容器逃逸，官方发布修复版本
- **确认度：高**（opencontainers/runc GitHub Release + BleepingComputer 报道）
- **要点**：`CVE-2025-31133`、`CVE-2025-52565`、`CVE-2025-52881` 均涉及绕过 runc 对 `/proc` 写入限制，最终可能造成完整容器逃逸。修复版本包括 `runc 1.2.8`、`1.3.3`、`1.4.0-rc.3` 及后续版本。
- **攻击条件**：报道提到通常需要能启动带自定义 mount 配置的容器，可通过恶意镜像/Dockerfile 等路径触发；截至报道时未见野外利用报告。
- **运维含义**：个人服务器、CI runner、K8s 节点、Docker 主机都应检查底层 runc 版本；尽量使用 rootless/user namespace，限制不可信镜像与特权容器。
- **来源**：https://github.com/opencontainers/runc/releases/tag/v1.3.3 ；https://www.bleepingcomputer.com/news/security/dangerous-runc-flaws-could-allow-hackers-to-escape-docker-containers/

### 3. Docker Engine AuthZ 插件绕过漏洞 CVE-2026-34040：需升级到 29.3.1
- **确认度：中-高**（Cyera 研究披露；Docker 官方公告页未在抓取片段中列出该项，需继续跟踪官方安全公告）
- **要点**：Cyera 披露 Docker Engine 授权中间件在处理大于 1MB 的请求体时可能让 AuthZ 插件看不到完整 body，导致安全策略被绕过；攻击者可通过一个填充后的 HTTP 请求创建特权容器并访问宿主机文件。研究方称影响使用 Docker AuthZ 插件的企业/CI/CD/多租户平台，修复版本为 `Docker Engine 29.3.1`。
- **运维含义**：如果 Docker API 暴露给 CI、自动化 agent、管理平台或 TCP/TLS 网络入口，应立即核查 Engine 版本与 AuthZ 插件依赖。
- **来源**：https://www.cyera.com/research/one-megabyte-to-root-how-a-size-check-broke-dockers-last-line-of-defense ；Docker 安全公告页：https://docs.docker.com/security/security-announcements/

### 5. libfuse io_uring 相关缺陷影响 Linux/FUSE/部分 Kubernetes CSI 场景
- **确认度：中-高**（SecurityOnline 报道 + libfuse GitHub Release；CVE 细节来自媒体报道，官方 release 说明为“可能安全关键”）
- **要点**：SecurityOnline 报道 `CVE-2026-33179` 与 `CVE-2026-33150` 影响 libfuse `3.18.0` 至 `3.19.0-rc0` 的 io_uring transport，可导致 DoS、内存泄漏，严重情况下可能 UAF/潜在代码执行。GitHub release 说明 `fuse-3.18.2` 修复了 io_uring 启动错误路径的 UAF/NULL deref 与初始化队列的 NULL deref/内存泄漏。
- **运维含义**：仅在显式启用 libfuse io_uring transport 或相关 CSI/FUSE daemon 时优先级较高；传统 `/dev/fuse` 路径据报道不受影响。
- **来源**：https://securityonline.info/libfuse-io-uring-vulnerabilities-cve-2026-33150-cve-2026-33179/ ；https://github.com/libfuse/libfuse/releases/tag/fuse-3.18.2

## 对 OpenClaw / Hermes / 个人服务器最有用的 3 条

1. **先查容器运行时链路**：确认 Docker Engine、containerd/runc 版本；runc 至少升级到含修复版本（如 `1.3.3`），Docker Engine 若使用 AuthZ 插件或暴露 API，关注 `29.3.1`。
2. **限制自动化 agent 与 Docker API 权限**：OpenClaw/Hermes 若能调用容器构建、CI、远程 Docker API，应避免给 agent 直接特权容器能力；对 Docker socket、TCP API、CI runner 使用最小权限与网络隔离。
3. **Kubernetes 若存在则优先查 ingress-nginx/CSI/RBAC**：检查 ingress-nginx 版本；禁止普通用户创建任意 PV；审计 NFS/SMB CSI `volumeHandle/subDir` 中的 `../`；必要时收紧 PV 创建权限。

## 风险 / 机会

### 风险
- **容器逃逸风险集中爆发**：runc、Docker AuthZ、KubeVirt、libfuse/CSI 都指向同一类后果：容器/工作负载边界失效后触达宿主机或集群控制面。
- **自动化扩大 blast radius**：CI/CD、AI agent、运维机器人若持有 Docker API、K8s admin、云凭据，会把一次容器/插件绕过放大成服务器或云账号级事故。
- **存储层破坏风险**：K8s NFS/SMB CSI 路径穿越会影响备份、共享卷和持久化数据，风险不只是泄露，也包括误删与破坏。

### 机会
- **把 SBOM 与镜像准入纳入日常流程**：Docker 近期内容持续强调 SBOM/供应链治理，可把镜像构建、依赖扫描、签名与准入策略结合起来。
- **完善容器最小权限基线**：rootless、user namespace、禁用特权容器、限制 hostPath、限制 Docker socket 暴露，是今天多条风险的共同缓解面。
- **建立“KEV + 官方 feed”日常巡检**：CISA KEV、Kubernetes CVE feed、Docker/runc/libfuse GitHub release 可作为每日自动化资讯源，减少依赖二手报道。
