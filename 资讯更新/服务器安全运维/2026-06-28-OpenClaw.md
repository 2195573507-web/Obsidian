## 最重要 5 条

### 1. runc 三个高危容器逃逸漏洞仍是 Docker/Kubernetes 运维最高优先级

- 漏洞：CVE-2025-31133、CVE-2025-52565、CVE-2025-52881。
- 影响：runc 是 Docker、containerd、CRI-O、Kubernetes 等容器平台底层 OCI runtime。多个来源确认这些漏洞可通过不同路径绕过 runc 对 `/proc` 写入的限制，造成完整容器逃逸、宿主机写入或拒绝服务。
- 利用条件：通常需要能启动带有恶意镜像/特殊挂载配置的容器；BleepingComputer 引述研究称尚未看到公开在野利用报告。
- 修复：runc 1.2.8、1.3.3、1.4.0-rc.3 及后续版本包含修复；Docker Engine 28.5.2 已更新 runc 到 1.3.3。
- 运维要点：仅升级 Kubernetes control plane 不够，必须更新/轮换 worker nodes、Docker/containerd/runc，并重启容器引擎或替换节点。

### 2. Linux 内核 DirtyClone（CVE-2026-43503）公开利用链，云主机/K8s/CI runner 风险上升

- 漏洞：CVE-2026-43503，DirtyClone，属于 DirtyFrag 系列 Linux kernel LPE。
- 影响：公开文章称 JFrog 于 6 月 25 日发布工作利用链，可让本地用户在未完整打补丁的 Linux 系统上提权到 root，并可能绕过常规文件完整性监控，因为攻击修改的是页缓存中的可执行文件映像而非磁盘文件。
- 高风险场景：多租户云主机、Kubernetes 节点、CI runner、允许不可信代码执行的主机；Debian/Fedora 默认启用非特权 user namespace 的环境风险更高。Ubuntu 24.04+ 有 AppArmor namespace 限制但仍需关注发行版公告。
- 状态：此处主要依据二级媒体对 JFrog 研究的报道，建议以 JFrog、发行版安全公告、CVE/NVD/厂商内核包为最终准绳；完整修复版本需按发行版确认。

### 3. Docker Engine AuthZ 绕过 CVE-2026-34040：使用授权插件的环境应尽快升到 29.3.1

- 漏洞：CVE-2026-34040，Docker Engine 授权绕过，CVSS 8.8（来源称）。
- 影响：Cyera 研究称，Docker AuthZ middleware 对超过 1MB 的请求体处理存在问题，授权插件可能看不到真实请求体，而 Docker daemon 仍处理完整请求；攻击者可用单个填充 HTTP 请求创建高权限容器、挂载宿主机文件系统，从而获得宿主机敏感信息和 root 级访问能力。
- 受影响场景：依赖 Docker AuthZ 插件（OPA、Prisma Cloud、Casbin、自研插件等）做生产/CI/CD/多租户容器准入控制的环境。若 Docker API 未对外暴露、且不使用 AuthZ 插件，直接风险较低，但仍应升级。
- 修复：研究称 Docker Engine 29.3.1 已修复。该条来源来自研究厂商与媒体搜索结果，建议用 Docker 官方安全公告/发行版包进一步确认。

### 4. VoidLink：面向 Linux 云与容器环境的模块化恶意框架，强调长期潜伏、云凭据与容器横移

- 事件：Check Point Research 披露 VoidLink，一套面向 Linux、云和容器化环境的高级恶意框架。
- 能力：自定义 loader/implant/rootkit、30+ 插件、HTTP/HTTPS/ICMP/DNS C2、P2P/mesh 通信、云环境识别（AWS/GCP/Azure/阿里云/腾讯云等）、Kubernetes/Docker 检测、凭据收集、容器逃逸/横移模块、eBPF/LKM/LD_PRELOAD 等隐蔽能力。
- 状态：Check Point 表示未观察到真实感染证据，样本可能仍处于开发/商业化阶段；但能力方向与个人服务器/云主机威胁高度相关。
- 运维含义：Linux 云主机已是高价值目标；仅依赖防火墙不够，需要进程/网络/文件/凭据访问行为监控，尤其关注 metadata service、K8s secrets、Docker socket、异常 eBPF/LKM、异常 DNS/ICMP 出站。

## 对 OpenClaw / Hermes / 个人服务器最有用的 3 条

### A. 立即把“容器运行时版本盘点 + 节点轮换”加入日常安全检查

- 检查 Docker、containerd、runc 版本；优先确认 runc 是否达到 1.2.8/1.3.3/1.4.0-rc.3 或发行版等效修复版本。
- 如果使用 K8s：确认所有 worker node 已使用修复后的 OS image/runtime；托管 K8s 也要主动 reimage/rotate node，不能只等 control plane 更新。
- 对 OpenClaw/Hermes：若有插件、任务或 sandbox 依赖 Docker/容器隔离，不要把容器当作强安全边界；避免运行不可信镜像，尽量使用 rootless、userns、只读根文件系统、最小 capability。

### B. 个人服务器应重点降低“本地提权 + 容器逃逸”的组合风险

- DirtyClone 类 Linux kernel LPE 与 runc 逃逸叠加时，风险会从“容器内执行”升级为“宿主机 root”。
- 建议：跟踪发行版内核安全更新；对于 Debian/Ubuntu 可评估限制非特权 user namespace（需注意可能影响 rootless Docker、Chrome sandbox、部分开发工具）；不要让公网服务或 CI runner 执行不可信代码。
- 对 OpenClaw/Hermes：外部内容处理、自动化脚本、下载/构建流程要避免落到宿主机高权限执行；日志中关注异常 `unshare`、`nsenter`、`mount`、`ip netns`、`iptables/nft`、`modprobe`、`bpftool` 行为。

### C. 监控重点从“端口是否开放”扩展到“云凭据/metadata/K8s secrets/Docker socket 是否被触碰”

- VoidLink 与 TeamPCP/PCPJack 类云原生威胁共同指向：攻击者越来越重视 Docker socket、Kubernetes secrets、云 metadata、CI/CD token、SSH key、`.env`、Git 凭据。
- 建议增加低成本检测：
  - 审计 `/var/run/docker.sock` 访问；
  - 监控访问 `169.254.169.254` / 云 metadata endpoint；
  - 定期扫描 world-readable secrets、`.env`、SSH key 权限；
  - 记录异常 DNS/ICMP 出站、长连接 beacon；
  - 对备份做离线/异地/不可变副本，定期恢复演练。

## 风险 / 机会

### 风险

1. **容器边界不等于安全边界**：runc 与 Docker AuthZ 问题都说明“容器创建前/创建时/运行时”都有逃逸面。个人服务器上把陌生镜像当作隔离沙盒风险很高。
2. **本地提权漏洞会放大一切低权限入侵**：DirtyClone 若确认影响当前发行版，任何 Webshell、低权限服务账户、CI job 都可能变成 root。
3. **云原生恶意框架正在模块化**：VoidLink 这类框架即便未确认大规模在野，也预示未来攻击会更懂 K8s、Docker、云 metadata 和 EDR 规避。
4. **AuthZ/准入控制可能“看不见请求”**：Docker CVE-2026-34040 对使用授权插件的团队是一次提醒：策略引擎本身也需要 fail-closed 和回归测试。
5. **KEV 与边界设备仍需持续排查**：Ubiquiti、Cisco、Lantronix 等设备/管理平台常被忽略，但一旦暴露就是入口。

### 机会

1. **建立每日自动盘点基线**：自动采集 `docker version`、`containerd --version`、`runc --version`、kernel version、K8s node image，和安全公告做比对。
2. **把容器安全策略前移到供应链**：启用镜像签名、SBOM、镜像来源 allowlist、CI 扫描；减少运行时才拦截的压力。
3. **把最小权限落地成模板**：默认禁用 privileged、hostPID/hostNetwork、hostPath、Docker socket 挂载；对需要特权的任务单独审批和审计。
4. **低成本提升备份韧性**：关键 OpenClaw/Hermes 配置、Obsidian vault、数据库、密钥材料应有加密、异地、不可变/版本化备份，并做恢复验证。
5. **监控规则聚焦高信号行为**：例如 `/proc/sysrq-trigger`、`/proc/sys/kernel/core_pattern` 写入，异常 mount/symlink 行为，metadata endpoint 访问，K8s secret 批量读取，eBPF/LKM 加载。

## 今日建议动作清单

- [ ] 盘点所有服务器的 kernel、Docker、containerd、runc 版本。
- [ ] 若使用 Docker/K8s：确认 runc 已修复；必要时重启 container runtime 或轮换节点。
- [ ] 若使用 Docker AuthZ 插件：核对 Docker Engine 是否已到 29.3.1 或官方等效修复版本；未修复前限制 Docker API 访问。
- [ ] 检查是否存在暴露的 Docker API、Kubernetes dashboard/API、Redis、Ray dashboard。
- [ ] 检查 Docker socket 是否被容器挂载；尽量移除或改用受限代理。
- [ ] 跟踪当前发行版对 CVE-2026-43503 的内核修复状态；不能立刻修复时评估限制非特权 user namespace。
- [ ] 为 OpenClaw/Hermes 关键目录和 Obsidian vault 确认最近一次备份可恢复。
