## 最重要 5 条

### 1. runc 发布 1.2.8 / 1.3.3 / 1.4.0-rc.3，修复 3 个高危容器逃逸漏洞（已确认）
- 影响：CVE-2025-31133、CVE-2025-52565、CVE-2025-52881。runc 官方说明三者可通过不同方式绕过 runc 对 `/proc` 写入的限制，最终导致完整容器逃逸；涉及 maskedPaths、`/dev/console` bind-mount 竞态、LSM/procfs 写入重定向等。
- 受影响：官方称 CVE-2025-31133 与 CVE-2025-52881 影响所有已知 runc 版本；CVE-2025-52565 影响 runc >= 1.0.0-rc3。runc 1.1.x 及更早不再支持。
- 处置：升级到 runc 1.2.8、1.3.3 或 1.4.0-rc.3；优先启用 user namespace/rootless、避免运行不可信镜像、容器内非 root 与 noNewPrivileges。
- 来源：
  - https://github.com/opencontainers/runc/releases/tag/v1.3.3
  - https://github.com/opencontainers/runc/releases/tag/v1.2.8
  - https://github.com/opencontainers/runc/security/advisories/GHSA-9493-h29p-rfm2
  - https://github.com/opencontainers/runc/security/advisories/GHSA-qw9x-cqr3-wc7r
  - https://github.com/opencontainers/runc/security/advisories/GHSA-cgrx-mc8f-2prm

### 2. libfuse io_uring 路径修复两个可能安全关键的问题，Kubernetes CSI 场景需关注（部分未确认）
- 影响：libfuse 3.18.2 release note 写明修复两个 io_uring 问题：启动错误路径中的 UAF / NULL deref，以及 `fuse_uring_init_queue` 中的 NULL deref / 内存泄漏。
- 第三方报道将其对应为 CVE-2026-33150 与 CVE-2026-33179，并指出 Kubernetes CSI drivers 若启用 libfuse io_uring transport，可能面临 DoS 或潜在代码执行风险；CVE 编号与 CVSS 细节未在 GitHub release 页面中确认，因此标注未确认。
- 处置：如果系统或容器工作负载启用了 FUSE/io_uring，跟踪发行版包更新或升级 libfuse 3.18.2；审视 CSI/FUSE daemon 是否以 root 运行。
- 来源：
  - https://github.com/libfuse/libfuse/releases/tag/fuse-3.18.2
  - https://securityonline.info/libfuse-io-uring-vulnerabilities-cve-2026-33150-cve-2026-33179/

### 3. Helm v3.18.4 安全发布，修复 Chart.lock 符号链接导致本地代码执行问题（已确认）
- 影响：GHSA-557j-xg8c-q2mm / CVE-2025-53547。恶意 `Chart.yaml` 与被符号链接的 `Chart.lock` 组合，在 `helm dependency update` 时可能把攻击者控制内容写入如 shell rc 或脚本等会被执行的文件。
- 受影响：Helm 3.18.3 及以下；Helm SDK 中 downloader Manager 执行 update 时也受影响。
- 处置：升级 Helm v3.18.4；升级前检查 chart 中 `Chart.lock` 是否为 symlink；CI/CD 不处理未知 chart 或先做 symlink 检查。
- 来源：
  - https://github.com/helm/helm/security/advisories/GHSA-557j-xg8c-q2mm
  - https://github.com/helm/helm/releases/tag/v3.18.4

### 4. NVIDIA Container Toolkit / GPU Operator 被曝高危容器场景漏洞，GPU/AI 节点需重点核查（部分未确认）
- 影响：第三方报道称 CVE-2025-23266、CVE-2025-23267 可导致容器中权限提升、数据篡改、信息泄露或 DoS；涉及初始化 hooks、update-ldcache hook 与 CDI/符号链接场景。GitHub Advisory 搜索结果确认存在相关条目，NVIDIA 官方安全公告页面本次抓取返回 403，未能直接验证公告正文。
- 受影响：报道称 NVIDIA Container Toolkit <= 1.17.7、GPU Operator <= 25.3.0；v1.17.8 release 可访问但页面主要列出修复/变更，未直接写明 CVE。
- 处置：GPU 主机或 K8s GPU 节点优先核查 `nvidia-container-toolkit`、`libnvidia-container` 与 GPU Operator 版本，计划升级到 1.17.8 / 25.3.1 或跟随发行版安全包。

## 对 OpenClaw / Hermes / 个人服务器有用的 3 条

1. **容器运行时补丁优先级应高于普通应用升级**  
   OpenClaw/Hermes 若部署在 Docker/K8s 上，runc 漏洞是今天最值得优先处理的项。建议在维护窗口中核查 `runc --version`、Docker/Containerd 包版本，并确认发行版是否已回补 runc 1.2.8/1.3.3 修复。

2. **自动化流水线要增加 Helm chart 安全前置检查**  
   如果未来 Hermes/OpenClaw 使用 Helm 部署，CI 在执行 `helm dependency update` 前应拒绝 `Chart.lock` 为 symlink 的 chart，并尽量固定可信 chart 源。开发机和 CI runner 都应升级 Helm 3.18.4+。

3. **FUSE/CSI/GPU 节点属于“高权限插件面”，需要单独资产清单**  
   FUSE daemon、CSI driver、GPU Operator、NVIDIA Container Toolkit 都常以较高权限接触宿主机。个人服务器即使没有 K8s，也应记录是否安装了 FUSE/GPU 容器工具；如果没有用，减少安装面和服务面。

## 风险 / 机会

### 风险
- **容器逃逸链风险上升**：runc 三个高危问题都指向宿主机级影响，且默认 Docker/Podman/K8s 使用链路广泛。只要能运行容器或影响构建流水线，攻击面就很现实。
- **LSM/传统隔离并非充分缓解**：runc 官方明确提到 CVE-2025-52881 可绕过 LSM 标签，AppArmor/SELinux 在组合攻击下保护有限。
- **供应链入口继续成为热点**：Helm chart、容器镜像、GPU hooks、FUSE/CSI 插件都说明“部署工具本身”正在变成攻击入口。
- **远程运维系统是高价值入口**：CISA KEV 中 SimpleHelp OIDC 验签绕过虽非严格 24h，但对个人/小团队服务器尤其危险，因为远程支持工具常直连内网资产。

### 机会
- **建立每日自动核查清单**：把 `runc/containerd/docker/helm/nvidia-container-toolkit/libfuse` 纳入版本采集，生成简单差异报告。
- **推动 rootless / user namespace**：runc 官方反复推荐 user namespaces/rootless containers；个人服务器可优先在非关键服务上试点。
- **把“不可信镜像/Chart 禁止策略”固化到 CI**：对个人服务器而言，最省心的安全收益来自默认拒绝未知镜像源、未知 Helm chart、symlink 异常文件。
