### 3. Kubernetes 官方 CVE Feed 6 月 29 日更新：SMB CSI Driver 路径穿越 CVE-2026-3865

- **时间**：Kubernetes 官方 CVE 列表更新时间 2026-06-29 00:14:59 UTC。
- **影响面**：使用 `smb.csi.k8s.io` 的 Kubernetes 集群，且允许用户创建引用 SMB CSI Driver 的 PersistentVolume。
- **要点**：`subDir` 参数校验不足，攻击者可通过路径穿越导致 SMB 服务器上非预期目录被删除或修改。CVSS 6.5；修复版本为 CSI Driver for SMB `>= v1.20.1`。
- **风险**：对个人服务器/家庭实验集群影响取决于是否使用 SMB CSI；若有 NAS/SMB 挂载，误删风险值得重视。
- **建议**：升级 SMB CSI Driver；限制 PersistentVolume 创建权限；检查 PV 的 `volumeHandle` 是否含 `../` 等可疑片段。
- **来源**：
  - https://kubernetes.io/docs/reference/issues-security/official-cve-feed/
  - https://github.com/kubernetes/kubernetes/issues/138319

### 4. Kubernetes 官方 CVE Feed 中 ingress-nginx 配置注入风险仍需排查：CVE-2026-4342

- **时间**：Kubernetes 官方 CVE 列表在 2026-06-29 更新中列出；GitHub issue 修改时间显示为 2026-03-19（非 24 小时新增，作为持续高风险关注）。
- **影响面**：安装 ingress-nginx 的 Kubernetes 集群。
- **要点**：特定 Ingress annotations 组合可注入 nginx 配置，可能导致以 ingress-nginx controller 上下文执行任意代码，并泄露 controller 可访问的 Secrets。官方说明默认安装下 controller 可能可访问集群范围 Secrets。
- **风险**：如果 OpenClaw/Hermes 使用自建 K8s + ingress-nginx 暴露服务，这是高优先级排查项。
- **建议**：确认是否安装 ingress-nginx；升级到修复版本；审计 Ingress 资源中的 annotations 与 `rules.http.paths.path` 可疑内容；收紧 controller 的 Secret 权限。
- **来源**：
  - https://kubernetes.io/docs/reference/issues-security/official-cve-feed/
  - https://github.com/kubernetes/kubernetes/issues/137893

### 5. Ubuntu Azure 内核安全公告：本地提权/容器逃逸相关内核缺陷

- **时间**：搜索结果在过去 24 小时内返回；公告页面为 USN-8461-1（具体发布日期页面摘取未完整显示，按“近期/待确认”处理）。
- **影响面**：Ubuntu Azure kernel；涉及云 VM、容器宿主机、Kubernetes 节点。
- **要点**：公告列出 Dirty Frag、Fragnesia、ptrace race、AppArmor notification 等多个 Linux kernel 缺陷；其中 Dirty Frag/Fragnesia 可能导致本地提权或容器逃逸。
- **风险**：云上 Ubuntu 节点、容器宿主机、K8s 节点需要优先关注；个人服务器如不使用 Azure kernel，需按实际内核公告判断。
- **建议**：Ubuntu/Azure 节点检查 `unattended-upgrades` 与内核重启状态；容器宿主机安排维护窗口重启以加载新内核。
- **来源**：
  - https://ubuntu.com/security/notices/USN-8461-1

## 对 OpenClaw / Hermes / 个人服务器有用的 3 条

1. **容器宿主机优先补 Docker / containerd / BuildKit / runc 链路**  
   Docker Engine 29.6.1（2026-06-26，非 24 小时内但很相关）修复多个安全问题：恶意镜像可导致过量内存消耗；BuildKit 自定义 frontend 可构造请求禁用 Seccomp/AppArmor 保护。若 Hermes/OpenClaw 依赖 Docker 构建或拉取外部镜像，应把 Docker Engine、BuildKit、containerd 纳入同一轮升级检查。  
   来源：https://docs.docker.com/engine/release-notes/29/

2. **K8s/容器集群要把“谁能创建 Pod/PV/Ingress”当作安全边界**  
   本次 Kubernetes/GKE/containerd/CSI/ingress-nginx 相关公告共同指向一个模式：攻击前提常是“能创建 Pod、PV 或 Ingress”。个人集群即便用户少，也应避免把 CI/CD token、机器人 token、低信任 namespace 赋予过宽 RBAC。

3. **备份和监控侧重点：补“删除/逃逸/Secret 泄露”检测**  
   - 对 SMB/NFS/NAS 备份目录开启快照或不可变备份。
   - 监控 Kubernetes Ingress/PV 变更事件。
   - 对容器宿主机关注 OOM、异常 `kubectl logs`、异常 Pod 创建、unexpected host mount 等信号。

## 风险 / 机会

### 风险

- **容器逃逸与宿主机破坏风险仍高**：Linux kernel、containerd、Docker/BuildKit、CSI Driver 都出现与容器边界相关的问题。
- **“可信内部权限”风险扩大**：多个漏洞要求攻击者具备创建 Pod/PV/Ingress 的权限，但这类权限在个人自动化、CI/CD、测试集群里常被放得过宽。
- **存储层误删风险**：SMB CSI Driver 路径穿越会影响外部文件服务器/备份盘，可能绕过应用层备份逻辑。

### 机会

- **建立每日/每周补丁清单**：把 `docker version`、`containerd --version`、`runc --version`、`kubectl version`、内核版本与待重启状态纳入自动巡检。
- **最小权限 RBAC**：把 CI/CD、OpenClaw 自动化任务与人工管理员权限拆开；限制 `create pods`、`create persistentvolumes`、修改 ingress annotations 的能力。
- **备份防误删**：NAS/SMB/NFS 目录使用快照、只追加备份或对象锁，减少 CSI/脚本误删带来的连带损失。
