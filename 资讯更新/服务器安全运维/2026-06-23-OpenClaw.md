## 最重要 5 条

### 1. Google Cloud 发布两条 6 月 22 日安全公告：Cloud Logging 与 App Engine

- **Cloud Logging（GCP-2026-039，中危）**：Google Cloud 公告称，Cloud Logging 漏洞可能允许攻击者通过重新创建目标 Cloud Storage bucket，劫持 log sink 目的地，导致敏感日志持续路由到攻击者控制项目。
- **App Engine（GCP-2026-038，高危，CVE-2026-8934）**：Google Cloud 公告称，Google Cloud Console 中 App Engine 的 GetDashboardAppStats GraphQL 私有 API 操作曾泄露跨租户请求日志；官方说明“不需要用户操作，漏洞已修复”。
- **影响判断**：对依赖云日志、审计日志、集中日志导出的个人/小团队云环境尤其重要。日志本身可能包含 IP、路径、请求体摘要、错误堆栈或 token 线索。
- **建议动作**：检查 Cloud Logging sinks 的目标 bucket 是否仍归属正确项目；检查日志导出目的地 IAM；对日志中的敏感字段做脱敏；保留最近一周日志路由变更审计。
- 来源：https://docs.cloud.google.com/support/bulletins

### 2. Kubernetes 官方 CVE Feed 于 6 月 23 日更新，多项 2026 CVE 涉及 ingress-nginx 与 CSI 存储驱动

- Kubernetes 官方 CVE 列表显示最后更新时间为 **2026-06-23 07:12:14 UTC**。
- 新近条目包括：
  - **CVE-2026-3865**：CSI Driver for SMB 通过 `subDir` 路径遍历，可能删除 SMB 服务器上非预期目录。
  - **CVE-2026-4342**：ingress-nginx comment-based nginx 配置注入。
  - **CVE-2026-3864**：CSI Driver for NFS 通过 `subDir` 路径遍历，可能删除 NFS 服务器上非预期目录。
  - 多项 ingress-nginx 相关配置注入、auth-url 绕过、Admission Controller DoS 等 2026 CVE。
- **影响判断**：即使个人服务器不运行完整 K8s，也应关注 ingress-nginx、CSI、NFS/SMB 挂载这类“边界组件”。这些组件常被误认为只是基础设施胶水，实际权限很高。
- **建议动作**：若有 K8s/Ingress NGINX/CSI 使用，核对官方 issue；限制动态创建 Ingress、PVC、Pod 的 RBAC；避免让不可信自动化任务直接写入 Ingress 注解或存储 `subDir`。
- 来源：https://kubernetes.io/docs/reference/issues-security/official-cve-feed/

### 3. GKE containerd 多漏洞公告仍在更新，Ubuntu 节点补丁仍未完全就绪

- Google Cloud GKE 安全公告 **GCP-2026-037**（发布 2026-06-18，更新 2026-06-20）确认 containerd 多个漏洞影响 GKE 容器运行时。
- 涉及 CVE：CVE-2026-50195、CVE-2026-53488、CVE-2026-53492、CVE-2026-53489、CVE-2026-47262。
- 官方描述：攻击者若有创建 Pod 的权限，可能绕过 Kubernetes 安全边界，造成宿主机 compromise、缓存投毒、拒绝服务。
- Google 已列出 Container-Optimized OS 节点 1.30～1.36 的修复版本；**Ubuntu node images 的补丁仍在进行中**。
- **建议动作**：GKE 上升级 COS node pool 至官方列出的版本或更高；Ubuntu 节点短期收紧 Pod 创建权限；对不可信/关键工作负载使用 `imagePullPolicy: Always`；监控 checkpoint/restore 功能使用。
- 来源：https://docs.cloud.google.com/kubernetes-engine/security-bulletins

### 4. Docker Desktop 4.79.0 于 6 月 22 日发布，偏稳定性/日志/策略缓存改进

- Docker Desktop 4.79.0 发布日期为 **2026-06-22**。
- 重点改进：
  - Logs 视图增加复制、过滤预设、清空日志等能力。
  - 修复 VM idle wake 后 Docker API 偶发 500。
  - 修复残留 symlink 导致的 “Integrity issue detected” 误报。
  - 修复 Registry Access Management 策略下载，改为用户目录缓存并增加防篡改检测，避免 permission denied。
  - QEMU 更新至 v10.2.3。
- **影响判断**：不是紧急安全更新，但对桌面开发机、CI 本地复现和容器日志排障有价值。策略缓存防篡改与 registry policy 下载可靠性对供应链控制有帮助。
- **建议动作**：开发机可按正常维护窗口升级；如果近期遇到 Docker Desktop 启动、日志、策略下载问题，优先考虑 4.79.0。
- 来源：https://docs.docker.com/desktop/release-notes/

### 5. Microsoft Defender for Cloud 文档 6 月 22 日更新，6 月新增多项容器/K8s/多云安全能力

- Microsoft Defender for Cloud release notes 更新于 **2026-06-22**。
- 6 月新增/更新重点：
  - 6 月 18 日：Function Apps / Logic Apps API security posture management GA。
  - 6 月 17 日：cloud scopes 扩展支持 K8s namespace、K8s cluster、多云 registry/repository。
  - 6 月 3 日：Kubernetes misconfiguration enforcement 预览，可在 admission 阶段 audit/block 风险部署。
  - 6 月 2 日：EKS/GKE runtime-discovered container images 与 K8s node vulnerability assessment 扩展预览。
- **影响判断**：趋势是“运行时发现 + admission 阻断 + 多云统一资产范围”。个人服务器可借鉴思想：把资产分组、基线检查、部署前阻断做成自动化，而不是只靠事后告警。
- 来源：https://learn.microsoft.com/en-us/azure/defender-for-cloud/release-notes

## 对 OpenClaw / Hermes / 个人服务器有用的 3 条

### 1. 日志导出目标必须当成安全边界

- Google Cloud Logging sink 漏洞提醒：日志系统不是“只读记录器”，它可能持续泄露敏感元数据。
- 对 OpenClaw/Hermes：
  - 检查运行日志是否包含 token、cookie、私聊目标、webhook URL、节点 ID。
  - 日志归档目录、对象存储 bucket、备份仓库都应有明确 owner 和最小权限。
  - 对外发通知时避免把完整异常堆栈、密钥路径、内部 session key 写入聊天消息。

### 2. 容器/K8s 的“能创建 Pod”几乎等同于高风险权限

- GKE containerd 漏洞与 Kubernetes CVE feed 都强调：Pod 创建、Ingress 注解、CSI `subDir`、checkpoint/restore 等能力可以变成宿主机或数据面风险。
- 对个人服务器：
  - 不要把 Docker socket、Kubeconfig、rootful `docker` 权限交给不可信自动化。
  - OpenClaw/Hermes 若调用容器执行任务，优先 rootless、只读挂载、无 Docker socket、限制网络出站。
  - 对自动化任务分层：能读文件 ≠ 能创建容器 ≠ 能改反向代理。

### 3. 供应链与镜像可验证性值得纳入日常运维

- Docker Engine 29.6.0 新增 `GET /images/{name}/attestations`，可获取镜像上的 in-toto attestation（如 SLSA provenance、SPDX SBOM）。
- Docker Content Trust/Notary v1 退役迁移也说明旧签名链路在退出历史舞台。
- 对个人服务器：
  - 新镜像上线前记录 digest、SBOM、来源仓库。
  - 自动化部署尽量 pin digest，不只 pin tag。
  - 周期性扫描基础镜像，优先处理可利用/运行时可达漏洞，而不是机械清零 CVE 数。
- 来源：https://docs.docker.com/engine/release-notes/29/ ，https://www.docker.com/blog/docker-content-trust-retirement-and-migration-guidance/

## 风险 / 机会

### 风险

- **日志泄露风险上升**：云日志 sink、应用日志、诊断包、容器日志都可能成为敏感数据外流通道。
- **容器边界继续被削弱**：containerd、Linux kernel、CSI、Ingress 等底层组件漏洞说明“容器不是强安全边界”。
- **自动化权限过宽**：AI agent、CI runner、运维机器人若持有 Docker/K8s/云平台高权限，一旦被提示注入、依赖污染或任务误配置触发，影响半径会很大。
- **未确认攻击事件噪声较多**：搜索中出现多篇关于 Docker/K8s 被勒索或蠕虫攻击的转载，但本轮未能找到过去 24 小时内的一手官方确认，暂不作为确定事件。

### 机会

- **把日志与备份做成可审计资产**：明确日志 sink、备份仓库、对象存储 bucket owner，设置生命周期与访问审计。
- **引入 admission / preflight 思路**：即使不用 K8s，也可在部署脚本中加入“禁止 root、禁止 Docker socket、禁止特权容器、必须 pin digest”等检查。
- **利用 SBOM/attestation 能力**：Docker Engine 29.6.0 的 attestation API 代表镜像 provenance/SBOM 将更易自动化接入。
- **多云安全基线可借鉴**：Defender for Cloud 的 cloud scopes、K8s misconfiguration enforcement、runtime image discovery 可作为个人服务器资产分组和基线扫描设计参考。
