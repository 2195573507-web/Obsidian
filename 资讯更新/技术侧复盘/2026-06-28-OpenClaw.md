## 最重要 5 条

1. **容器与 AI 工作流暴露面是今日最高优先级。** 服务器安全日报把 runc 三个高危容器逃逸漏洞、Docker AuthZ 绕过、DirtyClone Linux LPE、VoidLink 云原生恶意框架列为核心风险；晚间补充又加入 Langflow CVE-2026-5027 被报道野外利用。共同结论是：个人服务器不能再把“容器/低代码 AI 工具/自动化工作流”视作天然隔离边界。

2. **AI Agent 生态正在从“模型能力”转向“接口、协议、上下文和评测”。** Google Interactions API、Slackbot MCP Client、Meta Astryx（未确认）、GitHub Trending 中的 agent 规范/记忆/自动化项目，以及 CORE-Bench 论文对可靠性、OOD、效率、scaffold、人机协作 uplift 的强调，都指向同一趋势：真正可用的 Agent 需要稳定接口、可读上下文、可审计工具链和非准确率指标。

3. **本地/自托管模型和浏览器自动化工具链值得跟踪，但升级要隔离验证。** Ollama v0.30.11 涉及 opencode、Claude Code、Codex drift detection、MLX/Vulkan/Qwen2.5-VL 等运行时修正；Puppeteer、Vercel Labs agent-browser、OpenHands 等也在强化浏览器自动化、CLI、MCP、多模型与沙箱体验。它们都与 OpenClaw browser/tooling 能力相关，但涉及底层 runner、浏览器二进制、模型切换和权限，应先在测试机 smoke test。

4. **Agent 安全不应只靠“系统提示 + 听话人格”。** 当天 AI 报告提到 sycophancy 检测/steering 与 refusal/persona 耦合研究：更迎合、更服从的 persona 可能削弱拒答边界。对 OpenClaw/Hermes 这类长期助手，安全策略要靠审批、权限分层、事实核查、反确认偏误测试和高风险工具回归用例，而不是只靠提示词。

5. **AI 应用基础设施本身正在成为安全治理对象。** Langflow、MLflow、Docker Model Runner、MCP/agent 工具、Slack/企业协作入口、低代码自动化平台都可能引入上传接口、插件、webhook、token、Docker socket、metadata service、K8s secrets 等高价值攻击面。日报应把“AI 工具链版本与暴露面”列为独立巡检项。

## 对 OpenClaw / Hermes / 个人服务器有用的 3 条

1. **建立每日轻量安全基线：容器运行时 + AI 工具链 + 暴露端口。**
   - 记录 kernel、Docker、containerd、runc、Langflow/MLflow/n8n/Dify/Novu 等版本。
   - 检查 Docker API、Docker socket、K8s API/dashboard、AI 工作流服务、上传接口是否公网暴露。
   - 对 runc/Docker/AuthZ/Langflow 等高风险项，优先升级或下线公网入口。

2. **Hermes monitor cards 可加入 Agent 运行质量指标。**
   - 除成功率外，记录 token/耗时、工具失败率、重试次数、人工接管次数、scaffold 依赖、OOD 任务表现、审批触发情况。
   - 这些指标比单次“答对/答错”更能发现浏览器自动化、工具调用、权限边界和提示策略的真实问题。

3. **把 Skills / wiki / experience cards 作为 Agent 可读上下文资产继续标准化。**
   - GitHub Trending、OpenSpec、设计系统/MCP、Google/Slack API 方向都说明“给 Agent 读的规范”会越来越重要。
   - 建议本地技能与经验卡继续补齐：适用条件、输入/输出、验证步骤、风险、来源、回滚方式，减少临场提示词依赖。

## 风险 / 机会

### 风险

- **容器逃逸 + 本地提权组合风险高。** runc、DirtyClone、Docker AuthZ 若叠加不可信镜像/CI runner/公网服务，可能从低权限执行升级到宿主机 root。
- **公网 AI 工作流服务风险上升。** Langflow 报道中的路径遍历/任意文件写入与自动登录、上传、插件、MCP 等能力组合后，容易扩大到 RCE 或横向移动。
- **信源质量不均。** 今日部分产品动态来自 Google News 聚合、二级媒体或未能直接抓取原文；Meta Astryx、Mythos 细节、部分小模型报道需继续标注“未确认”。
- **Agent 过度顺从风险。** sycophancy 与 persona/refusal 研究提示：长期助手越“顺”，越需要外部约束与审计。
- **供应链与许可证风险。** 自托管自动化平台、浏览器二进制、Agent CLI、MCP server 会引入下载、插件、凭据和 webhook 面；n8n/Novu 等还需单独核对许可证。

### 机会

- **把安全巡检自动化成可复用日报模块。** 容器 runtime、AI 工具链版本、公开端口、Docker socket、metadata endpoint 访问、备份恢复状态都适合做低成本自动采集。
- **OpenClaw 浏览器工具可参考 agent-browser/Puppeteer 方向。** refs 稳定性、可访问性树、二进制完整性校验、CLI 错误模型和 snapshot 诊断是明确改进点。
- **多模型/本地模型混合工作流继续有价值。** Ollama 与编码工具集成增强，小模型和私有编码场景受关注；适合用测试机验证“云端强模型 + 本地小模型”的成本/隐私/延迟平衡。
- **Agent 评测与监控可以产品化。** CORE-Bench 相关指标、人机协作 uplift、工具失败率、审批触发率可转化为 Hermes 的监控卡片和周报指标。

## 建议动作清单

- [ ] 盘点当前服务器 kernel、Docker、containerd、runc 版本，确认 runc 是否达到修复版本或发行版等效修复。
- [ ] 检查是否运行 Langflow/MLflow/n8n/Dify/Novu 等 AI/自动化服务；确认版本、认证、端口暴露、上传接口限制。
- [ ] 检查容器是否挂载 `/var/run/docker.sock`、hostPath、hostNetwork/hostPID 或 privileged；尽量移除或改为受限代理。
- [ ] 为 OpenClaw/Hermes 关键目录、Obsidian vault、数据库与密钥材料确认最近备份可恢复。
- [ ] 将 Hermes 指标扩展到工具失败率、重试次数、审批触发、人机接管和运行耗时。
- [ ] 后续在测试机验证 Ollama v0.30.11、agent-browser/Puppeteer 相关能力；不要直接在生产路径升级。
