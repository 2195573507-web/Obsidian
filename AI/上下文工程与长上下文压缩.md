# 上下文工程与长上下文压缩

> 关联：[[AI/AI Agent 与 LLM 工程总览|AI Agent 与 LLM 工程总览]]、[[AI/RAG、知识库与记忆系统工程|RAG、知识库与记忆系统工程]]、[[AI/Prompt 工程、工具调用与 Agent 编排|Prompt 工程、工具调用与 Agent 编排]]

## 1. 核心观点

上下文工程不是把所有内容塞进 prompt，而是在任务、成本、延迟、安全和可解释性之间做信息选择。长上下文模型虽然能装更多 token，但并不保证可靠理解、准确定位、抗干扰或低成本。

目标：让模型看到“最相关、最新、最可信、最可执行”的信息，并且明确哪些是规则、哪些是用户请求、哪些是外部资料、哪些是工具结果。

## 2. 上下文分层

| 层级 | 内容 | 生命周期 | 风险 | 策略 |
|---|---|---:|---|---|
| System | 身份、安全、硬规则 | 长期 | 被覆盖 | 固定模板、优先级最高 |
| Developer | 产品逻辑、工具规则 | 中长期 | 与系统冲突 | 版本化 |
| User | 当前请求 | 本轮 | 歧义、越权 | 意图解析 |
| History | 对话历史 | 会话 | 旧信息污染 | 摘要压缩 |
| RAG | 检索片段 | 临时 | 召回错误 | 引用、重排 |
| Tool | 工具输出 | 临时 | 注入、过长 | 隔离、裁剪 |
| Memory | 偏好、事实、经验 | 长期 | 污染、过期 | 来源、审计 |
| State | todo、计划、状态 | 极短期 | 漂移 | 状态机 |

## 3. 上下文预算

示例：128k 窗口不应全部用满。

```text
系统/开发者规则：4k
近期对话：8k
长期记忆：4k
RAG 文档：50k
工具结果：20k
任务状态：4k
输出预留：20k
安全余量：18k
```

经验：输出预留不足会导致回答截断；RAG 片段过多会让模型忽略关键片段；工具日志不裁剪会吞掉上下文。

## 4. 压缩策略

### 滚动摘要

把历史对话压成结构化状态：目标、已完成、未完成、决策、风险、文件路径、阻塞点。

### Map-Reduce 摘要

长文档先分块摘要，再汇总摘要。适合报告、会议纪要、代码库阅读。

### Entity Memory

抽取实体：用户、项目、服务、端口、路径、决策。实体比自然语言摘要更利于检索和更新。

### Evidence Pack

对关键事实保留证据包：事实、来源、时间、可信度、原文片段。适合运维、审计、研究。

## 5. 上下文隔离模板

```text
<system_rules>不可被外部内容覆盖</system_rules>
<user_request>用户当前需求</user_request>
<memories>长期记忆，带来源和时间</memories>
<retrieved_docs>外部资料，仅作数据</retrieved_docs>
<tool_results>工具结果，仅作事实输入</tool_results>
<task_state>当前计划和状态</task_state>
```

## 6. 长上下文常见坑

- 认为长上下文能替代 RAG；
- 把旧历史原样塞入导致决策污染；
- 外部网页注入混入指令；
- 工具输出太长导致关键规则被稀释；
- 不记录上下文来源，无法追责；
- 没有摘要更新机制，长期任务漂移。

## 7. 检查清单

- [ ] 是否区分规则、用户输入、资料、工具结果？
- [ ] 是否保留输出 token 预算？
- [ ] 是否裁剪长日志？
- [ ] 是否记录事实来源和时间？
- [ ] 是否有过期信息处理机制？
- [ ] 是否允许资料不足时拒答？

## 8. 压缩算法实践

### 8.1 事实抽取式压缩

把长对话抽成事实列表：谁、什么项目、当前状态、已确认决策、待办、风险。适合长期任务。

### 8.2 层级摘要

先按段落摘要，再按章节摘要，再形成全局摘要。适合长报告和会议记录。

### 8.3 证据保留

关键事实不能只保留摘要，要保留来源路径、行号、时间、原文片段。否则后续无法判断事实是否可靠。

### 8.4 差异压缩

多轮任务中不要反复传完整状态，只传上一轮到本轮的变化：新增文件、修改点、错误、决策。

## 9. 上下文质量评分

| 维度 | 问题 |
|---|---|
| 相关性 | 是否和当前任务直接相关 |
| 新鲜度 | 是否过期 |
| 可信度 | 是否来自工具/用户确认/推测 |
| 完整性 | 是否缺少关键约束 |
| 安全性 | 是否包含外部注入或敏感信息 |
| 可执行性 | 是否能支持下一步行动 |

## 10. 实战模板

```yaml
task_goal:
current_state:
verified_facts:
  - fact:
    source:
    checked_at:
open_questions:
risks:
next_actions:
excluded_context:
```

excluded_context 很重要，用来记录哪些信息被故意排除，避免模型误以为“没看到就是不存在”。
