# AI Agent 与 LLM 工程总览

> 关联：[[MOC|知识库导航]]、[[AI/Prompt 工程、工具调用与 Agent 编排|Prompt 工程、工具调用与 Agent 编排]]、[[AI/RAG、知识库与记忆系统工程|RAG、知识库与记忆系统工程]]、[[AI/多 Agent 协作、安全评测与生产运维|多 Agent 协作、安全评测与生产运维]]

## 1. 核心定义

LLM 工程不是“调用一次大模型 API”，而是围绕大语言模型构建稳定、可控、可评测、可观测、可持续演进的软件系统。它既包含 prompt、RAG、工具调用、记忆系统等 AI 特有能力，也包含传统软件工程中的权限、日志、测试、部署、回滚、监控和成本治理。

AI Agent 是一种带有自然语言接口、可调用工具、可维护状态、能根据反馈迭代执行的软件执行单元。它不是“有意识的机器人”，而是由模型、上下文、工具、策略和运行时共同组成的工程系统。

```text
Agent = LLM + Prompt + Context + Memory + Tools + Planner + Executor + Evaluator + Policy + Runtime
```

其中每一层都可能出问题：模型会幻觉，prompt 会冲突，context 会污染，memory 会过期，tool 会失败，planner 会过度规划，executor 会误操作，policy 可能缺失，runtime 可能无日志。成熟的 Agent 工程就是不断给这些不确定性加边界、验证和恢复机制。

## 2. 从 Chatbot 到 Agentic System

| 类型 | 能力 | 典型场景 | 主要风险 |
|---|---|---|---|
| Chatbot | 对话问答 | FAQ、客服、文本生成 | 幻觉、上下文遗忘 |
| Copilot | 辅助建议 | 写代码、写文档、分析 | 建议不可验证 |
| Workflow + LLM | 固定流程中嵌入模型 | 审批、抽取、分类 | 流程僵硬、异常分支缺失 |
| Tool-Using Agent | 动态选择工具 | 搜索、读文件、查系统 | 工具误用、权限过大 |
| Multi-Agent System | 多角色并行协作 | 研究、审查、排障 | 成本高、冲突合并难 |
| Agentic Platform | 多用户、多工具、可运维平台 | 企业 AI 助手、自动化平台 | 安全、审计、治理复杂 |

很多失败的 AI 项目，是把 Chatbot 当 Agent 用，把 Agent 当万能员工用，把生产系统当实验环境用。正确做法是按风险逐层增强能力：先问答，再 RAG，再只读工具，再低风险写入，再高风险审批。

## 3. LLM 应用分层架构

```text
入口层：Web / IM / API / Cron
  ↓
身份与权限：用户、角色、租户、数据范围
  ↓
任务理解：意图分类、风险分级、是否需要工具
  ↓
上下文工程：历史、记忆、RAG、工具结果、状态
  ↓
模型路由：选择模型、预算、fallback
  ↓
推理与编排：prompt、planner、tool calling、workflow
  ↓
执行层：工具、沙箱、API、文件、数据库、浏览器
  ↓
验证层：schema、规则、测试、事实校验、安全检查
  ↓
结果层：回答、报告、文件、变更、审计记录
  ↓
运维层：日志、trace、metrics、评测、反馈、成本
```

这个分层的价值在于：任何问题都能定位到某一层。例如回答胡编通常是 RAG/上下文/评测层问题；工具误删通常是 policy/权限/审批层问题；成本过高通常是模型路由/上下文预算/多 Agent 编排问题。

## 4. Agent 核心模块详解

### 4.1 Prompt

Prompt 是模型行为协议，不是咒语。它应定义角色、任务、输入来源、输出格式、拒答条件、工具规则和安全边界。生产系统里 prompt 必须版本化，任何修改都要能回滚，并通过黄金测试集回归。

### 4.2 Context

上下文包括系统规则、用户请求、历史对话、RAG 文档、工具结果、长期记忆、任务状态。上下文工程的关键不是多，而是准。错误的旧事实、无关日志、未标注来源的资料都会降低输出质量。

### 4.3 Memory

记忆保存长期偏好、稳定事实、项目状态和可复用经验。记忆必须可审计、可删除、可过期。把所有对话都写入长期记忆会导致污染；把 mutable facts 当永久事实会导致错误自动化。

### 4.4 Tools

工具让 Agent 影响真实世界。所有工具都应有参数 schema、权限等级、超时、错误返回、审计日志。写操作、删除、部署、数据库修改、防火墙和 SSH 变更必须经过人工确认。

### 4.5 Planner

Planner 把目标拆成步骤。好的计划应该可验证、可中止、可重排。坏计划会过度复杂、忽略风险、不断循环。生产 Agent 需要最大步数、失败阈值和 replanning 机制。

### 4.6 Evaluator

Evaluator 检查输出是否满足格式、事实、引用、安全和任务目标。它可以是规则、schema、测试、另一个模型或人工。不要只依赖模型自评。

## 5. 典型架构模式

### 5.1 RAG QA

适合企业知识库和文档问答。关键是检索质量、引用来源、拒答策略和权限过滤。

### 5.2 Planner-Executor

适合多步骤任务，如代码修改、运维排障、研究报告。Planner 负责拆解，Executor 执行工具，Verifier 验证结果。

### 5.3 Human-in-the-loop

适合高风险动作。Agent 先生成候选方案、影响范围和回滚路径，用户确认后执行。

### 5.4 Supervisor-Worker

适合多 Agent。Supervisor 分配任务、控制上下文、合并结果；Worker 专注某领域，不直接写共享状态。

### 5.5 State Machine Agent

适合审批、客服、工单。每个状态允许有限动作，降低不可控性。

## 6. 失败模式

| 失败 | 表现 | 根因 | 防护 |
|---|---|---|---|
| 幻觉 | 编造事实、路径、API | 无检索/无验证 | RAG、工具验证、拒答 |
| 工具误用 | 调错工具或参数 | 工具描述差、权限缺失 | schema、policy、审批 |
| 循环 | 一直重复同一步 | 无最大步数 | step limit、重复检测 |
| 上下文污染 | 旧事实影响当前任务 | 历史未压缩 | 记忆分层、过期 |
| 成本爆炸 | token 急剧增长 | 多 Agent + 长上下文 | 路由、预算、缓存 |
| 安全绕过 | 输出密钥或执行越权 | 注入/权限不足 | 内容隔离、策略层 |

## 7. 工程成熟度模型

| 级别 | 状态 | 特征 |
|---|---|---|
| L0 | Demo | 单 prompt，无日志 |
| L1 | 可用 | 有 RAG 或工具，人工观察 |
| L2 | 可维护 | 有结构化输出、错误处理、日志 |
| L3 | 可评测 | 有黄金集、回归、指标 |
| L4 | 可治理 | 权限、审计、成本、安全策略完善 |
| L5 | 平台化 | 多用户、多模型、多工具、多 Agent 编排 |

多数项目卡在 L1-L2：能演示，但不可评测、不可审计、不可长期运行。真正生产化至少要到 L3。

## 8. 设计检查清单

- [ ] 任务边界是否清晰？
- [ ] 哪些输入可信，哪些只是外部数据？
- [ ] 是否允许拒答？
- [ ] 是否需要工具？工具是否有权限分级？
- [ ] 是否有高风险审批？
- [ ] 是否记录 prompt/model/tool/token？
- [ ] 是否有黄金测试集？
- [ ] 是否能回放一次失败任务？
- [ ] 是否支持成本预算和降级？
- [ ] 是否支持记忆纠错和删除？
