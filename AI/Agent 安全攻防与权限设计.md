# Agent 安全攻防与权限设计

> 关联：[[AI/安全/Prompt 注入案例库|Prompt 注入案例库]]、[[AI/Prompt 工程、工具调用与 Agent 编排|Prompt 工程、工具调用与 Agent 编排]]

## 1. 核心原则

Agent 安全的核心不是“让模型答得安全”，而是“让系统即使模型被诱导也不能越权”。Prompt 是软约束，权限、沙箱、审批、日志、回滚是硬约束。

## 2. 攻击面

用户输入、RAG 文档、网页、邮件、代码注释、工具输出、长期记忆、第三方 API 都可能成为攻击入口。间接 prompt 注入尤其危险，因为恶意指令藏在看似可信的外部资料里。

## 3. 权限模型

| 等级 | 能力 | 策略 |
|---|---|---|
| R0 | 查询状态、读取公开文档 | 允许 |
| R1 | 写草稿、生成报告 | 允许并记录 |
| R2 | 修改项目文件、运行测试 | 任务范围内允许 |
| R3 | 重启服务、改 Nginx/UFW/SSH | 人工确认 |
| R4 | 删除数据、数据库写、密钥轮换 | 多重确认或禁止自动 |

## 4. Policy Engine

工具执行前必须做策略判断：路径是否允许、工具风险等级、用户是否授权、是否涉及敏感数据、是否可回滚。

## 5. 沙箱

Shell 和代码执行要限制工作目录、网络、CPU、内存、超时、环境变量和可写路径。不要把生产密钥传入不可信执行环境。

## 6. 审批卡片

高风险操作审批必须包含：操作、原因、影响范围、回滚方式、验证方式、是否涉及数据删除、是否有备份。

## 7. 红队测试

构建测试集：泄露系统 prompt、读取 .env、删除文件、伪造工具结果、RAG 注入、记忆污染、越权访问。每次工具权限变更后回归。

## 8. 事故响应

发现 Agent 越权：暂停相关工具、保留 trace、撤销变更、轮换可能泄露密钥、补 policy、加入回归测试。
