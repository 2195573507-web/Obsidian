# 现代软件工程实战总纲

> 关联：[[软件工程/FastAPI 后端工程实践|FastAPI 后端工程实践]]、[[软件工程/数据库设计与性能优化|数据库设计与性能优化]]、[[软件工程/测试与质量保障体系|测试与质量保障体系]]

## 1. 软件工程目标

软件工程不是写出能跑的代码，而是用可控成本持续交付可靠系统。核心目标：正确性、可维护性、可测试性、可观测性、可演进性、性能、安全和文档化。

## 2. 从功能到系统

完整系统包括：需求理解、领域建模、接口设计、数据库设计、代码实现、测试、部署、监控、反馈和迭代。只实现功能会导致扩展难、测试难、上线不稳。

## 3. 需求澄清

需求要回答：用户是谁、目标是什么、非目标是什么、核心流程、数据对象、权限、异常、验收标准。推荐每个复杂需求写背景、用户、目标、流程、数据、权限和验收。

## 4. 代码结构

推荐中型 Web 项目分层：api、schemas、services、repositories、models、core、tests。API 层负责 HTTP，Service 负责业务，Repository 负责数据，Core 负责配置日志安全。

## 5. 架构原则

架构不是图，而是约束。多数项目优先模块化单体，边界清晰后再考虑拆服务。不要过早微服务，也不要所有逻辑堆进一个文件。

## 6. 可维护性

命名即文档；函数只做一类事；副作用集中在边界；业务规则放在 Service；复杂决策写 ADR；每次故障补 FAQ。

## 7. 质量防线

格式化 → lint → 类型检查 → 单元测试 → 集成测试 → E2E → code review → CI/CD → 监控告警。质量不是上线前手动看一遍。

## 8. 性能优化

先指标，后优化。常见瓶颈：慢查询、N+1、缺索引、接口返回过大、同步外部 API、缓存缺失、日志过多、连接池不合理。

## 9. 文档体系

README、架构文档、接口文档、数据库文档、运维文档、ADR、测试说明、变更日志都是软件资产。
