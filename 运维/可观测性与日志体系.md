# 可观测性与日志体系

> 关联：[[运维/服务器运维 MOC|服务器运维 MOC]]、[[运维/服务器安全加固、备份、监控与应急响应|服务器安全加固、备份、监控与应急响应]]

## 1. 三大支柱

可观测性包括 logs、metrics、traces。日志回答“发生了什么”，指标回答“趋势是否异常”，链路追踪回答“一次请求经过哪里、耗时在哪”。

## 2. 日志规范

结构化日志字段：timestamp、level、service、request_id、user_id、route、status、duration_ms、error_code、trace_id。

不要记录：密码、token、cookie、完整 Authorization、证书私钥、未脱敏个人信息。

## 3. 指标体系

RED：Rate、Errors、Duration。USE：Utilization、Saturation、Errors。服务器关注 CPU、内存、磁盘、inode、IO、网络、进程、端口、证书、systemd failed、Docker health。

## 4. 告警原则

告警必须可行动。不要为每个 warning 告警。分级：P0 全站不可用，P1 核心功能故障，P2 风险升高，P3 信息。

## 5. 排障仪表盘

面板应包含：入口状态、HTTP 5xx、Nginx upstream error、容器重启、磁盘趋势、证书剩余、SSH failed、最新部署、systemd failed。

## 6. 日志保留

应用日志可按 7-30 天保留；安全日志保留更久；高敏日志加访问控制。Docker json log 必须 max-size/max-file。

## 7. 日志字段规范

生产日志应结构化，推荐字段：timestamp、level、service、env、host、request_id、trace_id、user_id、route、method、status_code、duration_ms、error_code、message、resource_id。

不要记录 password、token、cookie、Authorization、完整个人信息、证书私钥、数据库连接串。

## 8. 告警设计

告警要可行动。坏告警会让人麻木。每条告警应包含：影响范围、可能原因、首查命令、升级路径、静默规则。

| 告警 | 阈值示例 | 首查 |
|---|---|---|
| HTTPS 失败 | 连续 3 次非 2xx/3xx | curl + nginx status |
| 磁盘高 | >80% warning, >90% critical | df + du |
| Docker 重启 | 10 分钟内重启 >3 | docker logs |
| 证书过期 | <30 天 | certbot certificates |
| SSH 成功异常 | 非预期 IP accepted | auth log |

## 9. Trace

Agent 和 Web 请求都需要 trace。一次 Agent 任务至少记录：用户请求、模型调用、RAG 检索、工具调用、审批、最终输出。没有 trace 的 Agent 失败后无法复盘。
